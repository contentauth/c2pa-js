h2. TL;DR

{{packages/c2pa-node}}'s Node.js bindings are built on Neon, a Rust↔Node
addon framework that ties every release to a specific Node ABI and requires
this package to carry ~10 files of bespoke Rust glue. We propose replacing
it with [koffi|https://koffi.dev], a pure-JS FFI library, calling directly
into c2pa-rs's own public C API (the same API c2pa-python, c2patool, and
the C++/WASM SDKs already use) instead of a custom Rust addon.

We built this as a real, working proof of concept — not just a design doc.
On branch {{ale/koffi-poc}}, we deleted the Neon binding entirely (~2,754
lines of Rust and its whole build pipeline) and replaced it with a koffi
binding loading a plain shared library. The existing test suite runs
against it *unmodified*, and *37 of 69 tests pass outright*, with the
remaining failures narrowing down to one clear, well-understood gap (below)
plus a couple of deliberately-scoped-out features.

{note:title=Decision needed from the team}
How to close the one significant capability gap this migration surfaces
({{Builder.addAssertion}} and four related methods) — see "Decision
Needed" below. Everything else in this proposal is validated and ready to
build on.
{note}

h2. Background & Motivation

{{packages/c2pa-node}} wraps [c2pa-rs|https://github.com/contentauth/c2pa-rs]
using [Neon|https://neon-rs.dev], a framework for building native Node.js
addons in Rust. This works, but comes with ongoing costs:

* *Bespoke binding glue.* The package carries ~10 {{neon_*.rs}} files that
reimplement a Node-specific interface over c2pa-rs's Rust API by hand.
* *N-API version coupling.* The compiled {{.node}} addon is tied to Node's
native ABI — every Node version bump is a potential compatibility
question, and the package needs a real Rust toolchain (or prebuilt
binaries per platform) to install.
* *A simpler path already exists and is proven elsewhere.* c2pa-rs
publishes its own public, MIT/Apache-2.0-licensed C API ({{c2pa_c_ffi}}
crate → {{c2pa.h}} / {{libc2pa_c}}). This is exactly what c2pa-python
(ctypes) and c2patool (the reference CLI) already consume. Rebasing
c2pa-node onto the same C API via a pure-JS FFI library (koffi) isn't a
novel approach — it's catching up to what every other non-Rust C2PA
consumer already does, and it means this package carries *zero Rust code
of its own*.

{warning:title=One consequential trade-off, named up front}
koffi's native callback mechanism is _synchronous_ — it runs on the main
thread and cannot await real async work. The current {{CallbackSigner}}
API is typed {{(data: Buffer) => Promise<Buffer>}}, and a signer that
genuinely does async work (e.g. a network call to sign) cannot be bridged
through koffi at all. This is a real breaking change for any consumer
using that pattern, not an implementation detail — see "What We Validated"
below for exactly how this fails and why.
{warning}

----

h2. Proposed Solution

Replace the Neon binding with a koffi FFI binding over c2pa-rs's public
{{c2pa_c_ffi}} C API, covering the package's existing public surface:
{{Reader}}, {{Builder}}, {{LocalSigner}}, {{CallbackSigner}}, and
{{Settings}}.

{code:title=Architecture}
c2pa-node (TypeScript)                c2pa-rs's c2pa_c_ffi crate (public, via libc2pa_c)
-----------------------                ------------------------------------------------------
koffi binding + public types            c2pa Builder / Reader / streams
buffer & pointer lifetime management    LocalSigner / CallbackSigner (synchronous only)
                                         Settings
{code}

*What moves to Rust:* nothing new. {{Builder}}/{{Reader}} functionality
already lives in c2pa-rs today — the koffi rebase means reaching it
through the existing public C API instead of compiling a per-Node-version
Rust addon.

*What stays in JavaScript:* the koffi binding itself, the public
TypeScript types, and buffer/pointer lifetime management (koffi has no
built-in RAII, so the wrapper classes handle that explicitly).

h2. What We Validated

We didn't just design this — we built it. On branch {{ale/koffi-poc}}:

h3. The migration itself

* Deleted {{packages/c2pa-node/src/*.rs}} and {{Cargo.toml}} (~2,754 lines
of Neon glue) along with all Neon-specific build tooling ({{@neon-rs/cli}},
{{cargo-cp-artifact}}, the postinstall script, the {{build:rust*}} scripts).
The package now has *zero Rust code of its own* and loads a prebuilt
{{libc2pa_c}} at runtime via koffi.
* Rewrote {{Reader}}, {{Builder}}, and {{Signer}} against a new koffi
binding layer, *preserving the existing public class and method
signatures* wherever the public C API supports it — so the pre-existing
test suite could run against the new implementation completely unmodified.
* {{Settings}} needed *zero changes* — it was already pure TypeScript with
no native calls at all.
* {{Trustmark}} and the generic X.509 identity-signing path are stubbed
with clear "not implemented" errors rather than reimplemented (see "Scope
We Deliberately Left Out" below).

h3. Early technical spikes (before the full rewrite)

||#||What we tested||Result||
|1|Loading the library and calling a version function via koffi|(/) Works cleanly|
|2|Building a manifest and signing with a local key, including async dispatch off the main thread|(/) Signed successfully, off-thread call confirmed working|
|3|A JS callback invoked from native code, on both the main thread and a worker thread|(/) No crash, but *the callback always runs on the main thread and must return synchronously* — this is the root cause of the async-signer limitation below|
|4|Full streaming sign/read of a real JPEG through JS-implemented Read/Seek/Write callbacks|(/) Produced a validly signed asset|

h3. The async callback signer limitation (confirmed and now handled cleanly)

This is the one significant _behavioral_ change, and we tested it directly
rather than just reasoning about it. The current {{CallbackSigner}}'s
callback always returns a real {{Promise}} — and there is no standard
JavaScript mechanism to unwrap a Promise synchronously, even one that has
already resolved, without yielding to the microtask queue. Since koffi's
native callback is a blocking call that can't yield, a genuinely
asynchronous signer callback simply cannot be bridged through it.

We built and verified the failure path: calling {{Builder.signAsync()}}
with such a signer now fails with a *clear, specific error message*
explaining exactly why, instead of surfacing as an opaque native
signature-validation error. This is the correct, honest outcome — we did
not attempt a workaround, because there isn't a safe one. *This needs to
be flagged to any consumer relying on genuinely asynchronous signing
before a real cutover.*

h2. Decision Needed: Five Builder Methods With No Public C API

Nearly all 20 Builder test failures trace back to a single shared test
setup step calling one method ({{updateManifestProperty}}) — it's really
*one root cause showing up 19 times*, not 19 separate problems. The
methods affected are: {{addAssertion}}, {{addRedaction}},
{{getManifestDefinition}}, {{updateManifestProperty}}, and adding an
ingredient without an accompanying asset file.

*Why this happened:* the old Neon binding could call directly into any
public method on c2pa-rs's Rust {{Builder}} type, because Neon addons
aren't restricted to a C ABI — they link Rust directly. koffi (like any
FFI binding for a non-Rust language) can only reach functions that have
been explicitly exposed through the C API, and these five haven't been.

*Important context — this isn't a gap we created, and it isn't unique to Node:*
* The underlying Rust capability is real and long-standing: {{add_assertion}}
appears in c2pa-rs's own top-level documentation example, is used 76+
times in the crate's own test suite, and has existed since the original
V2 Builder design.
* *c2pa-python* — the official Python bindings, built by the same upstream
authors, also using a C-ABI binding (ctypes) over this exact same public
API — has the *identical gap*. Its {{Builder}} class has no equivalent
methods either.
* *c2patool*, the reference command-line tool (written in pure Rust, no C
ABI involved at all), never needed these methods because its entire
interface is "hand me a complete manifest as JSON upfront" — the same
pattern c2pa-python's own documentation recommends exclusively.

This is a structural limitation shared by every non-Rust C2PA binding
today, not something specific to koffi or to this migration.

h3. Option A — Remove these five methods from Node's public API

||Pros||Cons||
|Brings this package's capability set in line with every other non-Rust C2PA binding that exists today; removes a footgun where our own tests exercise capabilities no other language binding can reproduce; zero additional engineering work; no dependency on an external repository's timeline.|Breaking change for any consumer using the incremental {{Builder.withJson(x); builder.addAssertion(...)}} pattern (our own tests still exercise it); the redaction workflow also depends on two of these methods, so redaction support would need rework too — unless providing all assertions/redactions upfront in the initial JSON covers real usage, which is worth confirming before committing to this path.|

h3. Option B — Add the missing functions to c2pa-rs's public C API

||Pros||Cons||
|Restores full feature parity with the current Neon binding, with no private dependency of any kind; every other C-ABI consumer benefits too — c2pa-python would gain the same capability for free once merged; the required Rust change is small (there's an existing, near-identical function to use as a template).|Depends on an external open source repository's review and release cadence; until merged, this package either carries a patched fork (ongoing maintenance burden) or ships without these methods in the interim; someone needs to actually write and shepherd the upstream pull request.|

{note}
*This is a decision for the team* — and, since the relevant code is
public, potentially worth raising with the broader C2PA developer
community. It is the one open item blocking a real cutover decision.
{note}

h2. Scope We Deliberately Left Out (Not Blocked, Just Not Attempted)

* *Trustmark (invisible watermarking).* Depends on a separate Rust crate
entirely outside c2pa-rs's public C API. Not a limitation of this
approach — just not built yet.
* *The generic X.509/CAWG identity-signing path.* The public C API does
expose a relevant function, but marshaling its argument types through
koffi was left unfinished in this round. Also not fundamentally blocked.

Neither of these involves anything Adobe-specific or private — we
deliberately kept this proposal scoped to functionality that already
exists in the published c2pa-node package today, migrating it from Neon
to koffi without introducing new capability at the same time.

h2. Risks & Mitigations

||Risk||Mitigation||
|Manual memory management — koffi doesn't manage native pointer lifetimes for you|Thin, RAII-style TypeScript wrapper classes handle allocation/free consistently; c2pa-python's ctypes-based binding over this same API is a working reference for the pattern|
|Packaging the native library for every platform|c2pa-rs already publishes prebuilt binaries for this exact library — comparable burden to today's prebuilt Neon addon, following a proven fetch pattern|
|Reader validation depends on trust settings being loaded|Load trust settings for the current thread before reading, as documented|
|Loss of asynchronous "bring your own" JS signer callbacks|Accepted as a breaking change (see "What We Validated"); revisit if a real consumer surfaces a hard requirement|
|Five Builder methods with no public C API today|See "Decision Needed" above — this is the one item that needs a team call before a real cutover|

h2. Pros & Cons Summary

h3. Pros
* Real, measurable reduction in complexity: ~2,754 lines of Rust and an
entire native build/packaging pipeline removed, replaced by roughly 1,000
lines of TypeScript loading a prebuilt library.
* The read path and local/callback signing reach full parity with today's
Neon binding — proven by the existing test suite passing in full (or in
the one expected, now-documented exception).
* Errors are more specific by default for synchronous calls.
* Puts this package on the same footing as every other non-Rust C2PA
binding in the ecosystem, rather than a bespoke Rust-linked implementation
that behaves differently from everything else.

h3. Cons
* The five-method Builder gap is real and is the single biggest blocker to
a full cutover (see "Decision Needed").
* Trustmark and the generic X.509 identity path aren't ported yet (not
blocked on anything fundamental, just not done).
* Asynchronous JS callback signers are no longer supported — a genuine
breaking change for any consumer using that pattern.
* Failures on the asynchronous call path lose some error-message
specificity compared to the synchronous path, due to a threading
limitation in how c2pa-rs surfaces error details.

----

h2. Open Questions

# *Streaming vs. data-hashed embeddable signing* for large assets — how to
avoid repeated main-thread hops during chunked I/O.
# *Release packaging* — this proof of concept already removes the old Neon
build tooling entirely, but shipping {{libc2pa_c}} by default still needs
a concrete decision, most likely following c2pa-rs's own prebuilt-binary
release pattern.
# *What (if anything) replaces asynchronous callback signing* for
consumers who need it — accept the gap, or design something new.

----

h2. Recommended Next Steps

# Resolve the *Decision Needed* above (Option A or B for the five Builder
methods) — this is the one item genuinely blocking a go/no-go call.
# Decide on and publish a deprecation notice for the asynchronous JS
callback signer API, since it's a breaking change for any existing
consumer of that pattern.
# Once the above are resolved, cut {{packages/c2pa-node}} over to the
koffi binding for real and remove the superseded Neon code.
